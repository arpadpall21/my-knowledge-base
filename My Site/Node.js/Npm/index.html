NPX (Node Package Runner) (Node Package Execute)
npx command@version - 1) checks the specified command if it's available globally (if available runs it as a normal global command)
- 2) if it's not a global command checks if it's insalled locally (in ./node_modules/.bin) (if the package is locally found runs the module)
- 3) if the command is not available globally and locally, temporarily installs the package (latest version) and runs it (one time run only = package not installed!) NPX 5.2.0/insall as package

Notes :

Problem with NPM globally istalled packages (libraries) and binaries

- the problem was caused by the file system access permissions (the user has no privileges to access some path)

- the solution is to reconfigure the location where npm installs global binaryes and libraries

Step 1) configure the new path where npm installs libraries and binaries

- create a new ~/.npm-global directory

- configure npm to install libraries and binaries into the new directory with the npm config set prefix ~/.npm-global command

- after this npm will install binaries into ~/.npm-global/bin and libraries into ~./npm-global/lib/node_modules

Step 2) configure system variables so the new locations are found by the system

- add the following 2 lines at the bottom of the ~/.profile file (this file runs at system start and registers the following 2 system variables so they will be permanent system variables)

- export PATH=~/.npm-global/bin:$PATH (adds the new path to the PATH system variable, so binaries installed in ~/.npm-global/bin folder are found by the system)

- export NODE_PATH=~/.npm-global/lib/node_modules (this is where Node.js looks for globally installed packages)

Remember This :

- "extraneous" package = installed package which is not listed in the package.json file

- NPX is a great way to try older package version or even older node.js versions

- from NPM v5.0.0 the package.json and package-lock.json files are automatically updated

Description and Demonstration :

- NPM (Node Package Manager) is the built in package manager of Node.js, it can download, uninstal and update Node modules and other applications as well

- NPM has a large comunity which offers a wide range of packages for Node.js, but we can use other package managers if we want to

- NPM caches downloaded packages, so at first it always tries to install packages from the local cache (on the system there's one npm cache)

DEMO 
// --version | -v  --------------------------------------------------------------------    
    $ > npm --version                                                                  // current NPM version returned 
    
// help -------------------------------------------------------------------------------
    $ > npm help                                                                       // prints the NPM help log 
    $ > npm help install                                                               // redirects to the install command's help page  
    
// root  ------------------------------------------------------------------------------
    $ > npm root --global                                                              // returns the absolute path of the global 'node_modules' folder  
    
// install | i | add  -----------------------------------------------------------------
    $ > npm                                                                            // automatically installs all packages and their dependencies registered in the 'package.json' file
    $ > npm --production                                                               // automatically installs production packages and their dependencies registered in the 'package.json' file   
    $ > npm sass gulp --global                                                         // installs sass and gulp globally 
    $ > npm sass gulp --save-prod                                                      // installs sass and gulp as production package 
    $ > npm sass gulp --save-dev                                                       // installs sass and gulp as developement package 
    $ > npm sass gulp --save-optional                                                  // installs sass and gulp as optional package 
    $ > npm sass gulp --no-save                                                        // installs sass and gulp without registering them in the 'package.json' file   
        
// uninstall | remove | r | rm | un | unlink  -----------------------------------------
    $ > npm uninstall gulp                                                             // uninstalls gulp from the local project 
    $ > npm uninstall gulp --no-save                                                   // uninstalls gulp but does not update the 'package.json' file 
    $ > npm uninstall gulp --global                                                    // uninstalls gulp globally  
    
// prune ------------------------------------------------------------------------------
    $ > npm prune                                                                      // uninstalls all extraneous packages (packages what are not registered in the 'package.json' file)    
    $ > npm prune --production                                                         // uninstalls all developement packages but does not updates the 'package.json' file 
    
// update | upgrade | up --------------------------------------------------------------
    $ > npm update                                                                     // updates all production packages only 
    $ > npm update sass                                                                // updates the local sass package (wihout its dependencies) 
    $ > npm update sass --depth 5                                                      // updates the local sass package and its dependencies nested 5 depth    
    $ > npm update sass --global                                                       // updates the global sass package 
    $ > npm update -D                                                                  // updates developement packages only 
    
// list | ls | ll | la ----------------------------------------------------------------
    $ > npm list                                                                       // lists all local packages and their dependencies in a tree structure (extraneous packages sometimes listed sometimes don't [TESTED: 2020-01-22])    
    $ > npm ll                                                                         // prints a more detailed list 
    $ > npm list --depth 0                                                             // lists local packages, top level only (direct dependencies only)  
    $ > npm list --global --depth 2                                                    // lists global packages and their nested dependencies up to depth 2 
    $ > npm list --dev                                                                 // lists developement packages only  
    
// init -------------------------------------------------------------------------------
    $ > npm init                                                                       // opens the 'package.json' file creation walkthroug 
    $ > npm init --yes                                                                 // creates a 'package.json' file with default settings  
    
// test | start | run|run-script  -----------------------------------------------------
    $ > npm test                                                                       // runs the 'sripts.test' value from the 'package.json' file
    $ > npm start                                                                      // runs the 'sripts.start' value from the 'package.json' file
    $ > npm run myScript                                                               // runs the 'sripts.myScript' value from the 'package.json' file (custom script)   
    $ > npm run                                                                        // lists the available scripts in the 'package.json' file

NPX (Node Package Runner)

- NPX (Node Package Execute | Node Package Runner) is a command line tool comes with NPM after version 5.2.0, it allows us to run packages what are not globally installed

- NPX can

  • run locally installed pacakges on the command line
  • run not insalled packages (temporarily installs the package -> runs it on the command line -> package not available anymore) (great to try an older package version without installing it)
  • run codes from a Remote GitHub Branch [NOT TESTED: 2020-01-23]

DEMO 
// sass command available globally (installed globally) -------------------------------
    $ > sass                                                                           // runs sass   
    $ > npx sass                                                                       // runs sass (NPX doesn't do anything here because the command is available globally)    
    
// sass installed locally -------------------------------------------------------------
    $ > sass                                                                           // -! throws an error "command not available" because sass is locally insalled only!   
    $ > npx sass                                                                       // runs sass from the local "node_modules/.bin" directory 
    
// sass not insalled on the system ----------------------------------------------------
    $ > sass                                                                           // -! throws an error "command not available" 
    $ > npx sass                                                                       // temporarily installs sass (from the npmjs.com or from the npm cache) and runs it 
    $ > sass                                                                           // -! throws an error "command not available", sass is not insalled !   
    $ > npx sass@1.0.0                                                                 // installs and runs (temporarily) the specified sass version

The package.json / package-lock.json / npm-shrinkwrap files

- the package.json file is a sort of config file of the current package, it keeps track all sorts of package informations (configurations) and dependencies

- the package-lock.json file is automatically generated by NPM, it describes the current exact package dependency tree, so when the package is installed next time Node will install the same exact depenceny versions, the 'package-lock.json' file cannot be published!

- the npm-shrinkwrap.json file is basically the same as the 'package-lock.json' file but it can be published, if both the 'npm-shrinkwrap.json' and 'package-lock.json' is present the 'package-lock.json' file is ignored!

NPM Semantic Versioning

- NPM very well defines package (module) versions

DEMO 
    *      |  x      |  latest                                                         // [latest].[latest].[latest]    // latest version 
    4      |  4.x    |  ^4.2.1                                                         // 4.[latest].[latest]           // version 4 with the latest minor version and patch   
    4.2    |  4.2.x  |  ~4.2.1                                                         // 4.2.[latest]                  // version 4.2 with the latest patch version   
    4.2.1  |  =4.2.1                                                                   // 4.2.1                         // this exact version   
    
    >4.2.1                                                                             // higher than 4.2.1 version 
    >=4.2.1                                                                            // equal 4.2.1 or higher version 
    <4.2.1                                                                             // lower than 4.2.1 version 
    <=4.2.1                                                                            // equal 4.2.1 or lower version  
    
    4.2.1 - 5.2                                                                        // from 4.2.1 to 5.2.9           // the latest version between the specified range   
    >4.2.1 || <5.2.9                                                                   // combines version semantics    // this example does the same as the above version semantic
