Global Infrastructure

- Azure backbone -> Azure's global network

- Region -> Geographic location where one or multiple data centers reside

- a region might have only one data center (Azure doesn't work like AWS), at service deployment we can control where (or which availability zones) to deploy our servcies

- each region have a region pair for resiliency reasons, the pair region is located a the same geographic area at least 300 miles away

- setting up the pair region is the developer's responsibility

- Availability Zone -> an individual data center group within a region

- availability zones are for resiliency resons (redundant service availability), therefore availability zones exist at least in a group of 3 (numbered 1, 2, 3, ...)

- availability zones have low latency connections between them

Governing Azure Services

Hierarchy, Scope and Tags

- Azure Hierarchy -> Logical hierachy for resource management

- the Azure Hierachy is composed of 4 levels (Management Group -> Subscription -> Resource Group -> Resource)

- Management Group, Subscription and Resource Group are esentially logical containers providing hierarchical resource management

- Resource -> the service itself

- described as JSON template

- must be part of one (but only one) resource group

- resource groups can contain resources in different regions

- not all resources are available in all regions (except global services)

- Scope -> a hierarchy level with all its sub levels (ex: subsription scope means all its resource groups and all those resource groups' resources)

- all resources within a scope are affected by the scope's roles and policies

- Tag -> additional information (key value pair)

- can be assigned to a resource, resource group or a subscription

- used for enhanced service management, automation, etc...

- tags are not inherited in the hierarchy

Access Control

- Active Directory (Entra ID) -> Microsoft's identity provider for all sorts of Microsoft products (Azure, Outlook, Skype, Teams, etc...)

- a Tenant (also called directory) is an Active Directory instance in Azure (represents a unique organization or entity)

- basically it's an organization namespace for users, groups, and applications, it's responsible for managing identities, authentication, and authorization for these objects

- Security Principal -> one identity in Azure that can be:

- User

- Group

- Service Principal -> technical user

- Application Registration -> the app registration in Active Directory

- Enterprise Applicaton -> basically a list of service principals

- Managed Identity -> authenticates resource without credential management

- a resource that allows applications and services to authenticate to Azure resources without needing to manage credential (eliminates credential management like, connection strings)

- 2 types:

- system assigned -> automatically assigned on resource creation, its lifecycle is tied to the resource (created and destroyed with the resource)

- user assigned -> independenly created from any other resources, the user is responsible for its management (can be attached to multiple resources)

- Role -> describe actions that an assigned identity can do (ex: Contributor)

- in Azure managing access to resources is done by RBAC (Role Based Access Control)

- a role must be assigned to a Security Principal (app or person) and a scope (defines who and where)

- RBAC is an authorization mechanism not an authentication one (for authentication we mostly use managed identity)

- Policy -> limitation of resources (ex: location restriction) (built-in or custom)

- Can be assigned to a resource group or any higher scope (cannot be assigned to a single resource directly)

- Applying a policy to a scope will affect current and future resources only (but we can remeditate to apply newly assigned policies to already existing resources)

- We can define what action to take when the policy is violated (block, warn, etc...)

- Initiative -> a collection of policies

- Blueprint -> describes an Azure environment (used to replicate the same environment)

- describes role assignements, resources and policy assignements

- Resource Lock -> prevent actions on the specified scope

- can be assigned to subscription, resource group or resource scope

- at the time of writing this there are types of locks, CanNotDelete and ReadOnly

- locks can be managed by administrators only

Cost Management

- we should use the right services for our needs in order to save money (ex: a spot VM could be a cheap option for non essential computing)

- Cost Management -> resource where consts can be tracked and controled

- Azure Calculator -> Azure tool to calculate service pricing

Service Availability (SLA)

- Microsoft promises service uptimes (in percent per month), if if fails to do deliver its promise we'll get a refound on that service, this is called SLA (Service Level Agreement)

- for each service the SLA is documented (ex: VMs have a 99.99% SLA meaning 4m 22s downtime per month)

- the SLA can be calculated when using multiple services (ex if we have 2 required services the SLA formula is S1 * S2 / 100)

- there is no SLA for free services

Messaging Resources

- Event Hub -> high-throughput message broker that collects, transfomrs and stores large amounts of data

- can ingest millions of events per second

- retains data for days (1 to 7), so multiple services can read the same data from it

- supports multiple protocols (AMQP, HTTP, Kafka)

- each Event Hub contains several partitions, these are basically message channels where the order is garateed, however you need to specify a partition key in your message to make sure that they land to the same partition, otherwise messages are distributed in round-robin fashion across partitions

- the Event Grids Namespace is a logical container where several Event Grids live, it's basically provides, throughput config, access control and scallability for a group of Event Grids

Storage

- Storage Account account for the below 4 storage types

- Blob Storage -> file storage

- files are stored in Containers

- there are 3 availability tiers (speed VS price)

- Hot -> for frequently data access

- Cool -> for infequently data access

- Archive -> low availability but low price

- File Storage -> stores files like the Blob Storage but supprots SMB (Server Message Block) protocol (files are accessible remotely)

- files are stored in Shares

- Queue Storage -> queue accessible anywhere in the world via authenticated calls using HTTP(S)

- a single message size can be up to 64 KB, the queue size is the capacity of the storage account

- Table Storage -> NoSQL semi structured data storage

- designed for fast data access and high volumes of data

- the storage account contains tables

- tables contain rows (entities) size limit 1MB

- rows contain properties (up to 255)

- default row properties are PartitionKey, RowKey and Timestamp (the PartitionKey and RowKey are indexed for fast data access)

- CosmosDB works like table storage but offers much better global availability, speed and privacy features

- accessible by various query languages (SQL, MongoDB, Table Storage, Cassandra, Germlin)

- Disk Storage various (HDD, SSD) attachable storage form VMs

- Managed disk -> the underlying storage is automatically managed, it also has better availability and security than unmanaged disk

- Unmanaged disk -> we have to managed the underlying storage, more flexible than unmanaged disk

- SQL Databases:

- MySQL

- PostgreSQL

- Microsoft SQL Server services:

- SQL Database

- SQL Managed Instance -> enhanced SQL Database with full access from the OS perspective (designed for organization migration on-premise SQL Server workloads)

Network

- Virtual Network -> IPv4 or IPv6 logical network for resource isolation

- must reside in a single region

- under the same subscription any VM can be used in a virtual network (even VMs in different resource groups)

- vnet peering -> peering Azure Virtual Networks in different regions (throug Azure Backbone)

- does no transitive communication (ex: in serial a connection A -> B -> C, A cannot communicate with C)

- Network Interface -> Virtual network interface card (NIC) that connects a VM into a virtual network

- Public IP -> Public IP for in or outbound internet traffic

- Static -> fixed IP (for in and outbound traffic)

- Dynamic -> changes every time the associated resource is restarted (for outbound traffic only)

- SKU (Stock Keeping Unit):

- Basic -> Static or Dynamic

- Standard -> Static

- Network Security Group (NSG) -> controls the subnet traffic within the Virtual Network

- Application Security Group (ASG) -> controls the traffic of a group of servers

- we can create a group of servers within a subnet, ASG controls the netwrok traffic of this group

- Virtual Network Gateway (VPN Gateway) -> allows and controls the communication between the virtual network and on-remises

- communication between virtual network -> public internet | private network | other Azure virtual network (even in different region)

- Load Balancer -> load balances the traffic to multiple VMs or VM Scale Set

- public load balancer -> load balances public internet traffic to VMs

- private load balancer -> load balances traffic within a virtual network

- Application Gateway -> load balances web (HTTP) traffic

- fine grained load balancing for web applications

- Route Table -> subnet traffic routing

- each subnet has one, it's created with default settings when a subnet is created, but we can overide default settings

- Firewall -> Azure firewall on virtual network (monitors and controls traffic)

- DDoS protection plans -> Distributed Denial of Service protection of a virtual network

- can protect autoscaling group to generate hight costs, etc..

- SKU (Stock Keeping Unit):

- Basic

- Standard -> provides monitoring and machine learning for better protection

- Front Door (CDN) -> Azure's CDN service

- Express Route -> a direct connection of an on-premise network and Azure (private traffic that does not go through the internet)

- Bastilon -> provides secure, RDP and SSH access to virtual machines without exposing them to the public internet (doesn't need public IP)

- Application Insights -> Resource that performance monitor and diagnosis other resource(es) (by default stores data on its own resource)

- defines: whant to collect (CPU, memory, sys logs, etc...), how to transform (filter) and where to send the collected data
