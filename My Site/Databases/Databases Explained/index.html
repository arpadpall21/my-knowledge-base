Description and Demonstration :

- the Database is a collection of sturctured (or semi-structured) data

- the DBMS (Database Management System) is a system which accesses and manages the data stored in the database

- a database system or (database in short) = data + DBMS + application

- MySQL, MongoDB, etc... are basically database servers, they manage their own data

- we can connect to these DB servers and communicate with them through database query languages (SQL or NoSQL)

Cursor

- it's a session a temporary work area created in memory, its used to retrieve data from the database chunk by chunk in sequential manner

- same principal as video streaming, we only have the current portion what we watch not the whole video is downloaded

Indexing

- indexes are basically pointers to database records (ex: an SQL table that stores 100000 records an index points directly to the queried record(s) instead of searching through all of them)

- indexes significantly increase the READ and sort performance but at the expese of CREAD, UPDATE, DELETE performance because indexes must be processed and maintained by the database

Index types:

- default index - indexes records, so the lookup and sort operations are much faster

- full-text index - indexes full text, so searching in the text is much faster

Relational Database VS Document-Oriented Database

Data Storage Structure

- in Relational Databases the data is stored in tables, hence the name 'structured' (all rows in the table must have follow the same schema)

- every table must have a primary key that uniquely identifies its row (a record) in the database

- foreign keys in the table reference a primary key in the 'parent' table (this is how the relationship is described between tables)

- in Document-Oriented Databases the data is stored in documents, this model follows an object oriented structure (which is schemaless = data storage model can be structured freely)

- it's a "programing way" to store data, objects have key/value pairs, structured hierarchically and can be easly stretched when needed

Relation types between tables in SQL databases

- SQL is a 'relational' database system where tables 'relate' to each other, there are 3 main ways how 2 tables can relate to each other

- 1:1 (one to one) one record has relation with only one record in the other table

- 1:M (one to many) one record has relation with many records in the other table

- M:M (many to many) many records have relations with many records in the other table (for this we usually use a Linking table)

Scaling (Sharding)

- Database scaling is basically splitting the same dataset to multiple pieces

- Vertical Scaling = Upgrading a single sotrage device capacity (more CPU power, storage capacity, etc...)

- enhance capability -> scale up

- decrease capability -> scale down

- Horizontal Scaling = Adding additional storage devices to the same database system

- enhance capability -> scale out

- decrease capability -> scale in

- Horizontally scaling Relational Databases is somewhat problematic because the data is strictrly structured

- Horizontally scaling Object-Oriented Databases is easy because the data is schemaless

Database access Language

- database 'programming' languages do CRUD (Create Read Update Delete) operations in databases

- Relational Databases use SQL (Structured Query Language) which is a standard language to access relational databases (still SQL slightly changes depending which database implements it)

- Document-Oriented Databases use NoSQL (Not Only SQL), there's no standard NoSQL language to access document-oriented databases

Vector Databases

- Vector databases are designed to store and manage vector embeddings, they are optimized for semantic search (ex: "Car" semantically similar to "Vehicle")

- Data is transcoded into vector representations (embeddings) and stored in multi-dimensional space

- Because embeddings are stored in a multi-dimensional space, search algorithms can quickly find the closest vectors

- A vector database on its own does not store the data, but embeddings that hold metadata in order to reference the original data (however, there are some combo solutions, ex: Chroma)

Concepts:

- Vector embedding:

- A numerical representation of data (text, image, video, etc...) in a continuous vector space

- A vector captures the semantic meaning and relationships of a piece of data (ex: "holiday" and "vacation" will be close to each other in the vector space)

- Indexing:

- Refers to how the data is organized and stored in the database to optimize search algorithms (ex: HNSW, IVF, LSH, etc...)

- So reindexing a vector database is the process of rebuilding the structural "map" that allows the system to find neighbors quickly

- Chunking:

- Process of breaking down large pieces of data into smaller, manageable chunks for efficient storage and retrieval

- Ex: splitting a long document into paragraphs or sentences where each chunk holds additional metadata like chapter, page, section, etc.

- Tokenization:

- Breaking down text into smaller units (tokens) such as words, subwords, or characters to create more manageable and relevant vector representations (ex: "I love programming" -> ["I", "love", "programming"])
