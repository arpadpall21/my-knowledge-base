Description and Demonstration :

- an ER (Entity Relationship) Diagram is a conceptual view of a Database Schema (sort of visual map of a Database Schema)

Entities and Attributes

- Entities represent a real word object

- Weak Entities represents a real word object but cannot be identified by its attributes, they must be in some relation with other Entities in order to exist

- Attributes represent properties (pieces of informations) about the Entity

- Composite Attributes have further (nested) attributes

- Multi Valued Attributes can have multiple values

- Derived Attribute values are derived from other attributes in the Database

DEMO

- the Student Entity represents the students

- the Exam Weak Entity cannot exist on its own (There must be students who can take Exams)

- the student_id and gpa are Attributes (information about the Students)

- the name is a Composite Attribute (has further nested Attributes)

- the club is a Multi Valued Attribute (a Student can belong to multiple clubs)

- the good_student is a Derived Attribute (good or not can be derived from other attributes (like form the gpa))

Relationship between Entities

- the Relationship describes the relationship between entities

- the Relationship Attribute describes information about the relationship

Partitipation

- Partial Partitipation = not all Entities are involved in the relationship

- Total Partitipation = all Entities are involved in the relationship

Cardinality

- number of instances of an Entity from a relation that can be associated with the relation

- 1 : 1 One-to-One (One Entity in Entity Set A can be associated at most one Entity om Entity Set B)

- 1 : M One-to-Many (One Entity in Entity Set A can be associated with multiple Entities in Entity Set B)

- M : 1 Many-to-One (Multiple Entities in Entity Set A can be associated to one Entity in Entity Set B)

- M : M Many-to-Many (Multiple Entities in Entity Set A can be associated to multiple Entities in Entity Set B)

DEMO

- the Takes Relation describes that the student cant take the class and vice versa

- the grade Relationship Attribute describes information about the relation (students taking classes will get a grade)

- a Student Partially Partitipate the relation (one student can take multiple classes)

- the Class Totally Partitipate the relation (All Classes can be taken by one student)

- the 1 : M (One-to-Many) Cardinality means that one student can take Many Classes

Description about the above Diagram

- The Company is orginized into Branches. Each Company has a unique number, and a name

- The Company makes it's money by selling to Clients. Each Client has a name and a unique number to identify it

- The foundation of the company is it's Employees. Each Employee has a name, birthday, sex, salary and a unique number to identify it

- An Employee can work for one Branch at the time

- Each Branch will be managed by one of the Employees that work there. We'll also want to keep track of when the current manager startad as manager

- An Employee can act as a supervisor for other Employees at the branch, an Employee may also act as the supervisor for Employees at other branches. An Employee can have at most one supervisor

- A Branch may handle a number of Clients, however a single Client may only be handled by one Branch at the time

- Employees can work with Clients controlled by their Branch to sell stuff. If necessary multiple Employees can work with the same Client

- We want to keep track of how many dollars worth of stuff each Employee sells to each Client they work with

- Many Branches will need to work with Suppliers to buy invertory. For each supplier we'll keep track of theri name and the type of product the're selling the Branch. A single Supplier may supply products to multiple Branches
